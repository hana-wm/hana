Read the following sources and use their findings on quality commenting guidelines in order to go through my repomix codebase and analyze the different files, their code and their comments (or lack of, or excessive amount of). Then make the necessary changes to these according to the guidelines, with the ultimate objective of improving the commenting throughout all files to the highest possible achievable grade of quality.

https://mitcommlab.mit.edu/broad/commkit/coding-and-comment-style/ (pay special attention to this one, as it's very thorough)
https://daily.dev/blog/10-code-commenting-best-practices-for-developers
https://medium.com/@sarveshgorwadkar/best-practices-for-writing-meaningful-code-comments-a-technical-guide-4131f3c9ece1

Work entirely on the directory that's facing the user, so that even if your response generation gets interrupted, the user will always be able to download all the artifacts you worked on up to that point. Work on an agile manner; the moment you spot a single change you could make, apply it right then and there instead of looking for more changes; once you applied it, continue searching for more changes. Do not write devlogs, and if you spot any devlogs (comments that comment on the developing process instead of the code itself) remove them, or rewrite them so that they aren't devlogs anymore.
